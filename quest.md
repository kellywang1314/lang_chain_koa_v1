#####  1. 在处理ai聊天的流输出时，fetch api和xhr api有什么区别？
- 读取方式
    - fetch ：原生支持 ReadableStream ，可用 response.body.getReader() 按 chunk 读取
    - XHR ：只能用 onprogress 监听并手动解析 responseText 增量
- 中断/取消
    - fetch ： AbortController 原生支持，易取消流
    - XHR ：用 xhr.abort() ，但控制粒度较粗
- 响应解析
    - fetch ：可以边读边解码，适合 SSE/Chunked
    - XHR ：拿到的是累积字符串，增量需要自己切分
- 错误处理
    - fetch ：只有网络错误才抛异常，HTTP 4xx/5xx 要手动判断 res.ok
    - XHR ： onerror / onreadystatechange 等事件更分散
- 可扩展性
    - fetch ：可配 ReadableStream 、 TextDecoderStream ，更适合 AI 流式处理
    - XHR ：历史方案，写起来更啰嗦

#####  2. 针对 AI 聊天界面的长列表，你会如何优化性能以保证滚动流畅？
常用优化策略
    - 虚拟列表 ：只渲染可视区域 + buffer（如 react-window / react-virtualized ）
    - 分段渲染 ：历史消息分页加载，首屏只加载最近 N 条
    - 消息高度缓存 ：避免重复测量导致 reflow
    - 图片/富文本懒加载 ：消息内图片延迟渲染
    - 减少重渲染 ：消息组件 memo ，局部状态隔离
    - 避免频繁 setState ：流式输出节流/批量更新（比如 16ms/50ms flush）
    - 合并 DOM 结构 ：避免深层嵌套与大量动态样式
    - 减少阴影/滤镜 ：这些会显著影响滚动性能
流式聊天特殊优化
    - 流式输出只更新最后一条消息
    - 按 chunk 合并更新 ，而不是每个 token 触发 re-render

#####  4. 在处理 LLM 流式输出（Streaming）时，前端如何优雅地处理 Markdown 解析和代码高亮？
核心思路 ：处理流式 Markdown 的关键是增量解析和状态管理，避免每次都重新解析整个文本：
- 增量解析：
不要等到整个响应结束才解析，而是在每一块数据（chunk）到达时，就对其进行增量解析。
使用支持增量解析的 Markdown 解析库，如 marked，或自己维护一个解析状态机，跟踪当前是否在代码块、列表、粗体等状态中。
- 代码高亮：
识别到代码块（```language）的开头时，初始化一个代码高亮器（如 Prism.js 或 Shiki）。
将后续的代码行逐行喂给高亮器，实时生成高亮的 HTML。
当代码块结束（```）时，完成高亮并渲染最终的代码块。
- 虚拟列表结合：将解析后的 Markdown 元素（段落、代码块、列表项）作为独立的列表项，结合虚拟滚动技术，即使是非常长的 AI 输出也能流畅渲染。

常用库组合
- Markdown： marked / markdown-it
- 高亮： highlight.js / prismjs
- React：搭配 memo + 局部状态，减少父组件重渲染

##### 5. 什么是 Prompt Injection（提示词注入）？前端在安全方面能做哪些工作？
- 定义
    - Prompt Injection 是指攻击者通过在正常输入中插入恶意指令，来改变 LLM 行为的攻击方式
- 影响
    - 可能导致信息泄露、数据损坏、系统崩溃等严重后果
- 前端安全措施
    - 输入过滤与校验：对用户输入进行长度限制和敏感词过滤。
    - 检测并阻止明显的注入模式，如包含 “忽略之前指令”、“你是一个” 等关键词的输入。
- 输出隔离：
    - 永远不要将用户输入直接拼接到系统提示词中，应将其作为独立的 user 消息发送给 LLM。
    - 对 AI 输出进行 HTML 转义，防止 XSS 攻击。
- 权限与沙箱：
    前端作为展示层，不直接执行 AI 生成的代码或命令。所有 Tool Call 都应通过后端服务进行，后端负责权限校验和安全沙箱。
- 用户教育：在 UI 中明确提示用户，AI 可能会被误导，对于重要决策不要完全依赖 AI 输出

#####  6. 管理 AI Agent 的多轮对话上下文（Context）
在项目中，管理多轮对话上下文通常采用以下策略：
- 中心化状态管理：使用 Redux、Zustand 或 Jotai 等状态管理库，将对话上下文（messages 数组）作为全局状态。所有组件（聊天窗口、历史记录、Agent 配置）都从这个单一数据源读取和更新。
- 结构化消息格式：每条消息都采用标准化的结构，包含 role（user/assistant/system/tool）、content、timestamp、metadata（如引用的文档 ID、调用的工具）等字段，便于后续的上下文压缩和检索。
- 上下文压缩（Context Window Management）：
  - 滑动窗口：当上下文长度超过 LLM 的 Token 限制时，自动移除最早的 N 条消息。
  - 摘要压缩：调用 LLM 对早期的对话历史进行摘要，用摘要代替原始对话，保留关键信息。
  - 向量检索：将历史对话向量化，当新的用户问题到来时，先从历史对话中检索最相关的片段，再将其注入到上下文中，而非全部历史。
- 持久化与恢复：将上下文序列化后存储在 IndexedDB 或后端，支持页面刷新后恢复对话，或在不同设备间同步

###### 6. 在构建 AI Agent 时，前端如何实现 Tool Call（工具调用）的交互反馈？
- 状态可视化：
调用中：在 AI 消息气泡中显示 “正在调用工具：获取天气...” 的占位符和加载动画。
调用成功 / 失败：将工具返回的结果（或错误信息）以结构化的卡片形式嵌入到对话流中，而不是纯文本。
- 可追溯性：
展示 Tool Call 的详细信息，如调用的函数名、传入的参数、执行耗时。
对返回的结果，提供 “查看详情” 的入口，让用户可以展开查看完整的原始数据。
- 用户确认：对于高风险的工具调用（如发送邮件、扣款），前端应拦截并弹出确认对话框，获得用户明确许可后再执行。
- 错误处理：当 Tool Call 失败时，提供友好的错误提示，并给出重试或修改问题的建议，而不是让 AI 直接生成一个可能是幻觉的 “解决方案”。

#####  监控 AI 产品的性能指标
除了传统的 Web 性能指标（FCP, LCP, FID, CLS），AI 产品还需关注以下特有指标：
- TTFT (Time To First Token)：从用户发送请求到 AI 输出第一个字 / Token 的时间，这直接影响用户对 “响应速度” 的感知。
- TPS (Tokens Per Second)：AI 流式输出的速率，决定了内容生成的流畅度。
- 上下文处理时间：将历史对话和知识库信息处理并注入到 Prompt 中的时间。
- Tool Call 成功率与耗时：AI Agent 调用外部工具的成功率和平均耗时，是衡量 Agent 可靠性的关键。  
- 幻觉率：通过人工或自动化评估，统计 AI 生成内容中事实性错误的比例。
- 用户等待放弃率：用户在 AI 响应过程中主动关闭或离开页面的比例。

##### React Fiber 架构如何提升复杂 AI 应用的响应性
React Fiber 架构的核心革新是将同步、不可中断的渲染过程改造为可中断、可恢复的增量渲染过程，这对提升复杂 AI 应用的响应性至关重要：
- 时间切片（Time Slicing）：Fiber 将渲染工作（Reconciliation）分解为多个小任务单元。每个单元执行后，React 会检查是否有更高优先级的任务（如用户输入、AI 流式输出更新）需要处理。如果有，就暂停当前渲染，让出主线程，待高优任务完成后再恢复。这避免了 AI 生成大量内容时，主线程被长时间阻塞导致页面卡顿。
- 优先级调度：Fiber 为不同的更新赋予不同的优先级。例如，AI 流式输出的逐字更新可以被标记为低优先级，而用户的点击输入则是高优先级。这确保了即使在 AI 生成内容的过程中，用户交互也能得到即时响应。
- 可中断与恢复：渲染过程可以被中断并在之后恢复，而不是从头开始。这使得 React 能够在空闲时间处理 AI 生成的复杂 UI，而不影响关键交互。

##### 实现简单 RAG 前端流程的技术点
一个简单的 RAG 前端流程包含以下核心技术点：
- 用户输入处理：接收并预处理用户的问题。
- 检索模块集成：
    - 调用后端检索接口，将用户问题向量化后，在向量数据库中查找最相关的文档片段。
    - 前端展示检索到的 “参考来源”，增强回答可信度。
- Prompt 工程：将检索到的上下文片段和用户问题，按照预设的模板组装成最终的 Prompt，发送给 LLM。
- 流式输出渲染：接收并渲染 LLM 的流式回答。
- 引用标注：在 AI 回答中，自动识别并高亮引用了哪些检索到的文档片段，用户可点击跳转到原文。
- 反馈机制：提供 “回答不准确” 的反馈按钮，将用户反馈用于优化检索策略和 Prompt 模板。


##### AI 自动生成 UI 组件” 的技术方案
实现 “AI 自动生成 UI 组件” 可以分为以下步骤：
- 需求解析：
用户输入自然语言描述，如 “创建一个带图标的卡片组件，展示用户信息”。
前端调用 LLM，将需求解析为结构化的组件规格，包括组件类型、Props、样式、交互逻辑。
- 代码生成：
LLM 基于解析出的规格和项目的设计系统（如 Ant Design、Material UI），生成对应的 React/Vue 组件代码。
代码生成时需遵循项目的 ESLint 规则和代码风格。
- 实时预览：
前端使用 esm.sh 或自定义的沙箱环境，动态执行生成的代码，并在 iframe 中实时预览组件效果。
- 交互式调整：
用户可以通过自然语言对生成的组件提出修改意见，如 “把背景色改成蓝色”。
前端将修改意见发送给 LLM，LLM 生成新的代码，前端更新预览。
- 导出与集成：
用户确认后，将生成的组件代码导出为 .tsx 文件，并自动添加到项目的组件库中，供其他地方复用。


##### 应对 AI 响应不确定性（幻觉）的前端补救措施
前端无法从根本上消除幻觉，但可以通过设计降低其影响：
- 透明化来源：对 RAG 生成的内容，展示 “引用来源”；对 Tool 调用结果，标注数据更新时间，明确信息可信度。
- 分级提示风险：对高确定性内容，若无来源则提示 “未验证”；对低确定性内容，明确标注 “AI 生成，可能存在偏差”。
- 提供纠错入口：设计 “回答错误” 反馈按钮，用户提交的纠错信息可用于优化后端的 Prompt 和知识库。
- 技术层面校验：对接后端的事实校验接口，对 AI 输出的关键信息进行实时校验，校验失败则提示用户。
- 上下文兜底：缓存对话上下文，当检测到明显幻觉时，自动重置上下文并提示用户简化问题。
